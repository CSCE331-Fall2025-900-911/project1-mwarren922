Resi Low Cost Encoder

Redi Media, LLC is a software company that specializes in high-quality transmission for web, multisite, and on-demand streaming for churches. Resi sells video encoding and decoding hardware that interfaces with their web services.

In 2023, I worked on developing The Resi Mini, an Arm64-based SBC video encoder with a RockChip RK3588 chipset aimed at bringing low-cost 1080p video streaming to smaller churches. My primary contribution to this project was developing a Java application that encodes an HDMI signal and integrating it with Resi’s media upload infrastructure.

This application has two primary responsibilities: recording metadata and utilizing that metadata, along with user preferences, to encode an HDMI signal. I sourced the metadata by parsing V4L2 (Video for Linux 2) to get fields such as resolution, fields per second, and interlaced or progressive format. This information and the desired output format were fed into a GStreamer pipeline that produces stream chunks that can be uploaded to any desired destination.

Resi OS Migration

To replace CentOS 7, which was being used by Resi Media's x86 encoders, I created a custom and lightweight distribution of Debian 12, fine-tuned for video encoding/decoding, using Debian's live-build tool. I implemented git-like version control on this OS, using OSTree for seamless deployment of updates. I also developed and implemented a live-migration tool for porting systems from CentOS 7 to Debian 12 without any user interaction.

The OS Live migration replaces the old system while allowing a rollback until the new OS is operational. The first step is allocating space on a device to write the new operating system. Next, the bootloader is updated to recognize the new OS installation and boot into it. The new OS is built with a slew of tests that verify an operational system and only deletes the old system once they are complete.

Hardware Incident Resolution Platform

The Deprecation of CentOS-7 left many of Resi’s hardware in a non-updatable state. I devised a Python-based cloud worker that accessed every affected device via TeamViewer, navigated to a shell prompt, and injected a command to resolve the issue. The program used Tesseract OCR image detection to navigate the UI, logging various states of failure and success based on the virtual display output. The cloud worker could be triggered and log its results via Google Pub/Sub. This platform resolved the issue on 1000+ encoders, saving hundreds of hours of manual support work and provided a mechanism to deal with similar incidents, should they occur.
